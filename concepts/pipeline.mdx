Каждый скан в chifu проходит через два этапа. Цель второго этапа —
запускать только те инструменты, которые имеют смысл именно для
вашего стека, а не жечь время на заведомо бесполезных проверках.

## Этап 1. Recon

**Запускается всегда.** chifu должен понять, с чем имеет дело.

1. **Проверка доступности.** Сервис обращается к цели и убеждается, что
   она отвечает. Если цель недоступна — второй этап не выполняется, в
   результате будет только информация о недоступности.
2. **Авто-апгрейд HTTP → HTTPS.** Если вы указали `http://`, а сервер
   отвечает на `https://`, chifu прозрачно переключится на HTTPS. В
   recon будет флаг `httpsUpgraded`.
3. **Passive scan.** За одну-две секунды chifu сам, без запуска внешних
   инструментов, читает заголовки, `robots.txt`, `sitemap.xml`,
   `.well-known/security.txt` и ищет в теле известные сигнатуры (ключи,
   токены, утечки). Сюда же входит **обнаружение anti-bot challenge**:
   если сайт за Vercel BotID / Cloudflare Bot Fight / Akamai / DataDome
   возвращает challenge-страницу, chifu это видит и помечает как
   medium-finding (см. [Firewall bypass](/dashboard/firewall-bypass)).
4. **Recon-инструменты параллельно.** Краулер, port scan, fingerprint
   технологий, поиск поддоменов, fuzzing коротких путей. Описание всех —
   на странице [Tools](/concepts/tools).

По итогам этого этапа chifu знает: какой сервер, какая CMS (если есть),
какой WAF/CDN, какие технологии и их версии, какие URL с параметрами
есть на сайте, какие формы, какие открытые порты.

## Этап 2. Vuln

Запускается только если цель достижима. Здесь chifu проверяет
уязвимости, **прицельно** — на URL и формы, найденные в первом этапе, а
не «на главную страницу и надеемся».

### Почему некоторые инструменты пропускаются

Это нормально и сделано специально. На странице скана вы увидите
такие записи рядом с tool'ом:

| Что вы видите | Что это значит |
|---|---|
| `skipped: target behind CDN` | Сайт за Cloudflare/Vercel/Akamai. Сканировать порты edge-узла — бесполезно, это не ваш сервер. Пропускается `nmap`. |
| `skipped: static SPA` | Сайт — современный фронтенд (Next.js, Nuxt, Astro и т.п.) без серверного бэкенда. Инструменты для legacy-стеков (поиск `/wp-admin`, `joomscan`, CGI-проверки nikto) не дадут ничего, кроме шума. |
| `skipped: needs param URL` | Не нашлось ни одного URL с параметром в query string — sqlmap и dalfox нечего проверять. |
| `skipped: needs CMS=joomla` | joomscan запускается только на Joomla-сайтах. |

Если вы хотите явно проверить, что какие-то инструменты не пропускаются
неоправданно — посмотрите блок **Recon** на странице скана: если chifu
обнаружил CMS, backend-стек или формы, vuln-этап будет шире.

### Прицельный fan-out

- **sqlmap** запускается до 6 раз за скан: до 3 раз на самые
  интересные URL с параметрами + до 3 раз на найденные POST-формы.
- **dalfox** работает по тем же URL для XSS.
- **nuclei** получает список из топ-50 целей (главный URL, URL с
  параметрами, найденные fuzzing'ом пути, JS-эндпоинты) и проверяет их
  по шаблонам критических/высоких/средних/низких уязвимостей.

## Лимиты

- **15 минут** — максимум на один скан целиком. Обычно скан укладывается
  в 5–10 минут.
- **До 5 инструментов одновременно** — баланс между скоростью и
  нагрузкой на цель.
- Каждый инструмент имеет свой собственный таймаут (обычно 60–600 с).
  Если он не успел — частичный результат всё равно попадает в скан, а в
  таблице tool'ов будет помечено как timed out.

## Что попадает в результат

| Раздел страницы скана | Откуда |
|---|---|
| **Risk gauge** и **Score breakdown** | [Композитная оценка риска](/concepts/risk-score) |
| **Tech stack** | Recon — обнаруженные технологии с версиями и confidence |
| **Known vulnerabilities** | CVE-lookup по версиям из recon + находки nuclei |
| **Other findings** | Все остальные находки от sqlmap, dalfox, nikto, passive scan, … |
| **Tools** | Сырой вывод каждого запущенного / пропущенного инструмента |
