---
title: "Firewall bypass"
description: "Per-domain секрет, который chifu подставляет в каждый запрос, чтобы Vercel / Cloudflare WAF пропускал сканер."
---

Современные сайты часто стоят за Vercel BotID, Cloudflare Bot Fight
Mode, Akamai Bot Manager или DataDome. Эти системы отдают chifu
challenge-страницу вместо вашего сайта — и сканер в итоге проверяет
challenge-форму вместо реальной поверхности атаки. Score такого скана
будет низкий, а confidence — `low`.

**Firewall bypass** решает это: chifu отправляет в каждый HTTP-запрос
дополнительный заголовок с секретом, а вы у себя на CDN добавляете
custom-правило: «если этот заголовок такой-то — пропускать без
проверки».

<Warning>
  Bypass работает только на ваших собственных доменах. На странице
  [Domains](/dashboard/domains) сначала подтвердите владение — пока
  домен в `Pending`, секция Firewall bypass недоступна.
</Warning>

## Включить bypass

1. **Dashboard → Domains**, раскройте карточку верифицированного домена.
2. Прокрутите до секции **Firewall bypass** — снизу будет три вкладки
   с провайдерами: **Vercel**, **Cloudflare**, **Custom**.
3. Нажмите **Enable firewall bypass**.

chifu сгенерирует случайный секрет (формат `chf_b_...`) и сохранит его
для домена. Секрет видно в поле **Header value** (по умолчанию скрыт
точками; иконка глаза показывает его). Имя заголовка — `x-chifu-bypass`
(см. поле **Header name**).

С этого момента chifu автоматически подставляет эту пару во все запросы
к вашему домену и его поддоменам — и в passive scan, и во все
recon-инструменты, которые поддерживают custom-заголовки. Дополнительно
включать ничего не нужно.

## Настроить custom rule на CDN

chifu только шлёт заголовок — само правило на CDN вы создаёте сами.
Дашборд показывает пошаговый рецепт под каждый из трёх провайдеров.

<Tabs>
  <Tab title="Vercel">
    1. В проекте на vercel.com → **Settings → Firewall → Custom Rules**.
    2. **Create Rule**:
       - **IF**: Request Header `x-chifu-bypass` **Equals** `<ваш секрет>`
       - **THEN**: `Bypass`
    3. Сохраните правило.

    Действие `Bypass` пропускает Bot Protection, OWASP Core Ruleset и
    Attack Mode — то есть весь стек защиты Vercel за один шаг.

    <Note>
      Vercel Bypass — это не то же самое, что **Deployment Protection
      bypass** (preview / auth-gated deploys). Для preview-окружений
      есть отдельный механизм с переменной `CHIFU_VERCEL_BYPASS` для
      self-hosted сценариев; в облаке chifu используется обычный
      custom-rule bypass.
    </Note>
  </Tab>
  <Tab title="Cloudflare">
    1. На дашборде Cloudflare выберите домен → **Security → WAF →
       Custom rules**.
    2. **Create rule**:
       - Expression (в expression-editor): `(http.request.headers["x-chifu-bypass"][0] eq "<ваш секрет>")`
       - Action: **Skip** → отметьте **All managed rules** (или конкретно
         WAF managed rules + Bot Fight Mode, в зависимости от вашей
         конфигурации).
    3. Сохраните.
  </Tab>
  <Tab title="Custom">
    Если у вас другой WAF (Akamai, AWS WAF, F5, modsecurity) — создайте
    правило, которое **пропускает / whitelist'ит** запрос при наличии
    заголовка `x-chifu-bypass` с указанным значением.

    Общий принцип:
    - совпадение должно быть **точным** по значению,
    - правило должно стоять **перед** правилами Bot Management,
    - действие — `allow` / `skip` / `bypass` (терминология
      провайдера).
  </Tab>
</Tabs>

## Управление секретом

В блоке Firewall bypass на странице домена:

- **Copy** рядом с **Header value** — скопировать секрет в буфер
  обмена.
- **Rotate secret** — сгенерировать новый секрет. Старый перестанет
  работать; нужно обновить значение в custom rule на CDN.
- **Remove** — отключить bypass для домена. chifu перестаёт слать
  заголовок, custom rule на CDN можно удалить (или оставить — оно
  просто не сработает).

## Скоуп

- Секрет привязан к **корневому домену** и применяется ко всем его
  **поддоменам** автоматически.
- Если для конкретного поддомена нужны **другие** заголовки (например,
  поддомен живёт за другим CDN) — раскройте карточку поддомена и
  настройте bypass там отдельно.

## Безопасность

- Секрет показан только владельцу организации. Другие участники
  команды видят пустое поле — это контролируется RLS-политиками.
- При **редиректе на чужой хост** chifu **снимает** заголовок bypass —
  ваш секрет не утечёт в логи стороннего сервиса, на который случайно
  ведёт цепочка редиректов.
- Меняйте секрет (**Rotate secret**), если подозреваете, что он попал
  не туда, куда нужно.

## Как понять, что bypass нужен

Сигналы на странице скана:

- В блоке **Other findings** есть запись типа **«Anti-bot challenge
  intercepted»** (severity medium, tool `passive`, type
  `scan-limitation`).
- **Risk score** = 1 с **confidence: low**, в `Tech stack` — пусто или
  только CDN.
- Большая часть инструментов в блоке **Tools** имеет 4xx/403/429
  ответы.

После включения bypass перезапустите скан — confidence и score должны
прийти в норму.
